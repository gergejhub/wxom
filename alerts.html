Alerting Guide — tiles, tags and “why” explanations (OM-aligned)

This page documents every tile, trigger rule, score/color logic and OM-derived policy tag used by the WX Monitor. The goal is that nothing is a black box: when an airport turns red or a tile lights up, you can trace back what triggered and why it matters operationally.

Important: this is an advisory monitor. It does not replace the OM/SOP, commander decision-making, performance calculations, actual RWY-in-use, RWYCC/SNOWTAM, ATIS/NOTAM, airport operator reports, or the official briefing chain. Use it for early detection and fast situational awareness.

1) How to read the dashboard

Basics

WX Monitor is a network weather monitor: each table row is one airport/station. The system parses the raw METAR and TAF text and extracts the signals that matter operationally (VIS, RVR, ceiling, gusts, weather phenomena). From these it builds: tiles (a fast summary), tags (what triggered and why), and a severity score.

METAR vs TAF in this monitor:
  • NOW tiles are METAR‑primary: they aim to represent what is happening now.
  • FORECAST tiles are TAF‑only and live in a separate panel, so the “now” picture stays clean.
  • The weather display under the tiles (raw METAR/TAF + decoded view) is intentionally unchanged.
  • The table’s Worst visibility is intentionally conservative (designed to highlight potential risk quickly).

The main UI elements

  • Tiles: “what should I look at first?”. Clicking a tile filters the table to airports currently triggering it.
  • Table tags: compact “why” labels. They are source-coded as M (METAR), T (TAF) or M+T (both).
  • Drawer (airport details): click a row. The section Minima / Policy (OM) – Why? shows evidence (matched tokens, RVR groups, computed crosswind, inferred RWYCC bucket, etc.).
  • Views: AUTO (normal desktop) vs TV (large display). Notifications behave differently — see section 4.6.
False-positive protection (important):
Weather phenomena are detected from the WX tokens in the report body. The monitor explicitly strips report headers (including the ICAO station code) before scanning, so airports like LGTS or GCTS cannot accidentally trigger the TS tile just because their ICAO code contains “TS”.

2) Colors & alert levels

Meaning

The dashboard uses a compact, consistent color language. Tiles and rows are not “decisions” — they are signals that a condition is present and may require attention. Always confirm with the raw METAR/TAF and local operational context.

Level What it generally means Typical examples
OK Normal conditions, no meaningful operational risk indicated by the monitor. Good VIS/RVR, no significant phenomena, moderate wind.
MED Noticeable operational “noise”. Worth a glance; may impact handling/capacity. G25, light snow, marginal ceiling, moderate vis reduction.
HIGH Meaningful risk; likely to influence tactical choices (alternates, turnaround plans, capacity). G30+, snow + poor VIS combo, convective risk, LVO environment.
CRIT Severe conditions or multiple severe signals at once. Requires immediate attention and verification. Very low VIS/RVR, heavy precip prohibitions (OM), extreme gusts, compounded hazards.
Pillar escalations: The main “severity score” can be escalated by dedicated pillars (e.g. wind and snow). This is intentional: a single parameter crossing a threshold may be operationally critical even if the “generic” score would otherwise stay lower.

3) Scoring, VIS/RVR/CIG and “Worst” logic

Details

The monitor calculates two separate signal sets: one from the current METAR and one from the TAF. Tiles are split accordingly: NOW tiles are METAR‑primary, while FORECAST tiles are TAF‑only.

3.1 What goes into the severity score

  • Visibility & RVR: low VIS/RVR bands contribute strongly. If RVR is present, it is treated as the stronger signal.
  • Ceiling / cloud: low ceiling increases risk; CB/TCU identification can elevate the score.
  • Weather phenomena: TS / freezing precipitation / heavy precipitation / blowing snow increase risk.
  • Wind gust: gust thresholds can escalate the final alert level (see WIND tile and pillar rules).
  • Compound effect: multiple medium signals can create a HIGH/CRIT score even if no single tile looks dramatic.

3.2 Why “Worst visibility” can look worse than METAR VIS

The table’s “Worst visibility” is intentionally conservative. It is designed as a risk spotlight: it may take the minimum over reported RVR groups and/or include forecast elements (depending on the column implementation), while the NOW tiles focus on the current observation (METAR‑primary).

Rule of thumb: Use the tiles to decide where to look, then confirm in the row details: raw METAR/TAF, decoded values, and the Drawer evidence (Minima / Policy (OM) – Why?).

3.3 Source coding

Tags on the airport cards indicate their source: M (METAR), T (TAF), M+T (both). This is critical when a “forecast risk” exists but the current observation is still acceptable.

4) Tiles, tags, triggers and OM context

Rules

Tiles provide an “at-a-glance” view of common operational weather risks across the network. The monitor separates NOW (METAR) and FORECAST (TAF) so that current conditions do not get mixed with longer-range uncertainty. In the table, tags are source-coded as M / T / M+T.

Anti false-positive guard (TS/CB and ICAO codes):
Before scanning for WX phenomena, the monitor strips report headers (including ICAO station codes). This prevents airports such as LGTS or GCTS from triggering the TS tile just because the ICAO code contains “TS”.

4.1 NOW (METAR) tiles

ENG ICE OPS Priority FZFG + extremely low VIS (now)

Trigger: METAR contains FZFG and the reported visibility ≤ 150 m.

Why: “freezing fog + very low visibility” is operationally high-impact; this tile is treated as a hard priority signal.

Note: this is a monitoring rule (early warning), not an OM minima decision by itself.

CRITICAL CRIT METAR severity score ≥ 70 (may escalate by pillars)

Trigger: METAR severity score ≥ 70.

Why: typically indicates multiple severe signals in combination (low VIS/RVR, convective, low CIG, heavy precip, strong gusts).

Audit: open the airport Drawer → check “Summary/Triggers” and the OM “Why?” panel to see what contributed.

VIS/RVR < 300 LOW VIS METAR visibility or minimum RVR

Trigger: METAR visibility < 300 m or METAR minimum RVR < 300 m.

Why: this band commonly correlates with LVO environment and capacity/operational risk.

Rule: if RVR is present, it is treated as the stronger signal; VIS is not converted into RVR.

TS/CB WX Convective WX token (TS / VCTS); CB/TCU elevates score

Tile trigger: a TS‑type WX token in METAR, e.g. TS, -TSRA, +TSGR, VCTS, etc.

CB/TCU: cloud groups like BKN020CB / SCT030TCU are detected. They can increase the score and appear as tags, but do not necessarily raise the TS tile on their own unless a TS/VCTS token is present.

Protection: ICAO station codes cannot trigger TS (no “LGTS/GCTS” false positives).

WIND GUST METAR gust ≥ 25 kt

Trigger: METAR maximum gust ≥ 25 kt.

Why: from G25 it becomes meaningful operational noise (handling, turbulence, alternates, crosswind risk).

Pillar escalation: the Wind pillar can escalate the alert level independently: 25→MED, 30→HIGH, 40→CRIT.

SNOW WX METAR SN / SHSN / BLSN

Trigger: METAR contains SN, SHSN or BLSN (blowing snow).

Why: snow can quickly degrade VIS/RVR and turnaround performance. The Snow pillar escalates with the combined effect (snow + visibility + wind), not snow alone.

Extra: BLSN is treated as a high‑impact snow indicator (often paired with drift and very poor VIS).

4.2 FORECAST (TAF) tiles

FORECAST tiles are derived only from the TAF and appear in a separate panel, so they do not clutter the “now” picture. Treat them as “heads-up” planning signals.

CRITICAL (TAF) CRIT TAF score ≥ 70 (forecast)

Trigger: TAF severity score ≥ 70.

Why: forecasted combination of severe signals. Confirm with time groups (TEMPO/PROB/BECMG) and local context.

VIS/RVR < 300 (TAF) LOW VIS Any forecast VIS/RVR value below the band

Trigger: any VIS or RVR value in the TAF drops below 300 m in any time group.

Why: forecast LVO environment can affect planning (alternates, capacity). Always check the time window.

TS/CB (TAF) WX Forecast TS/VCTS; CB/TCU recognition

Trigger: forecast TS/VCTS WX tokens in the TAF time groups.

CB/TCU: cloud groups (BKNxxxCB, SCTxxxTCU) elevate the score and may appear as tags.

WIND (TAF) GUST TAF gust ≥ 25 kt

Trigger: forecast gust ≥ 25 kt in the TAF.

Why: planning signal; consider timing, direction and crosswind exposure.

SNOW (TAF) WX TAF SN / SHSN / BLSN

Trigger: forecast SN/SHSN/BLSN in the TAF time groups.

Why: heads-up for snow/ice ops; combine with forecast wind and vis bands.

4.3 OM policy tags (airport card tags, not tiles)

OM-derived items are shown as tags on the airport cards (in the table and Drawer), not as separate tiles (to avoid clutter). Tags appear only when the relevant threshold is crossed. The Drawer panel Minima / Policy (OM) – Why? provides the evidence.

TO PROHIB (METAR) OM OM-A heavy precipitation prohibition list (advisory)

Trigger: METAR contains a token from the OM-A “takeoff prohibited” heavy precipitation list (e.g. +SN, FZRA, GR, etc.).

What OM means operationally: for specific intense phenomena, takeoff is prohibited. This monitor detects the relevant tokens in the raw report and flags it.

Evidence: Drawer → Minima / Policy (OM) – Why? → “matches”.

OM context: OM-A “Heavy precipitation” section (e.g. 8.3.8.7). Always validate the current revision and local procedures.

LVTO (METAR) OM Reference RVR/VIS < 550 (advisory)

Trigger: reference low-visibility value < 550 m (RVR if present, otherwise VIS).

What OM means: LVTO environment can require specific procedures/constraints. The monitor tags additional sub-bands (see 4.4) for clarity.

Evidence: the Drawer shows which RVR group/VIS value drove the band.

XWIND (METAR) OM OM-B RWYCC-based crosswind limit (estimated + best RWY)

Trigger: computed crosswind component exceeds the estimated limit derived from the OM-B RWYCC table.

How it works: the monitor selects the “best” runway heading (from runway data) to compute crosswind from the METAR wind, then compares to a limit. Because RWYCC is not available from METAR/TAF, the RWYCC bucket is inferred from WX tokens (advisory).

Evidence: Drawer shows wind, chosen runway heading, computed xwind, estimated RWYCC bucket, and the applied limit.

VA (METAR) OM Volcanic ash detected

Trigger: METAR contains VA (volcanic ash).

Why: high-impact hazard. Treat as immediate attention and cross-check with official SIGMET/VAAC information.

TO PROHIB (TAF) OM OM-A heavy precipitation prohibition list (forecast)

Trigger: TAF time group contains a heavy-precip prohibition token (same list as METAR case).

Why: heads-up that a prohibition condition is forecast in a time window.

LVTO (TAF) OM Forecast reference RVR/VIS < 550

Trigger: in any TAF time group, reference RVR/VIS drops below 550 m (RVR if present, otherwise VIS).

Why: planning signal. Always check the time group (BECMG/TEMPO/PROB) and expected duration.

VA (TAF) OM Volcanic ash forecast

Trigger: TAF contains VA in any time group.

Why: heads-up; confirm with official volcanic ash products.

4.4 Low visibility & landing category tags (threshold-only, hierarchical)

To avoid noise, these tags are shown only when a threshold is crossed and the system shows the most restrictive band per source (METAR vs TAF). Example: if RVR is 350 m, you get LVP (<400), not LVTO+LVP stacked.

RVR<125 OM Below absolute band (advisory)

Trigger: minimum reported RVR < 125 m.

Operational meaning: this is an extremely low band; treat as “below / near minima” depending on equipment and approvals. The monitor flags it as an immediate attention item.

LVTO<150 QUAL OM Very low RVR — crew qualification requirement (advisory)

Trigger: minimum RVR < 150 m (but not below 125 m).

Operational meaning: very low visibility takeoff band; typically requires specific crew qualification/procedure. The monitor flags it so you can validate against the applicable OM/approval.

LVP (<400) OM Low RVR — LVP in force / approval-type band

Trigger: reference RVR < 400 m.

Operational meaning: LVP environment (advisory label). Confirm with aerodrome LVP status and operator procedures.

LVTO (<550) OM LVTO band (RVR preferred, otherwise VIS)

Trigger: reference value < 550 m (RVR if present, otherwise VIS).

Operational meaning: LVTO environment. This is the broad band; the monitor will show stricter sub-bands when applicable.

RVR REQ (<800) OM VIS/CMV < 800 → RVR reporting expected (if missing)

Trigger: visibility (VIS/CMV) < 800 m and no RVR group is present in the report.

Operational meaning: in this regime, RVR reporting is expected/required for meaningful LVO operation. The monitor flags “missing RVR” so you can validate the report completeness and operational impact.

CAT2+ / CAT3 ONLY / CAT3<75 OM RVR bands (generalised thresholds, advisory)

Trigger: based on minimum RVR bands. Tags are shown only below thresholds (hierarchical per source).

  • CAT2+: RVR < 450 m (general “CAT II or better” environment indicator).
  • CAT3 ONLY: RVR < 175 m (very low band; CAT III‑type environment).
  • CAT3<75: RVR < 75 m (extreme band; treat as near/below minima depending on approvals).

Note: exact minima depend on aircraft/approach category, lighting, approval, and aerodrome capability. The monitor uses conservative generalised bands.

XWIND>limit + RWYCC<3 likely OM Crosswind table + inferred RWYCC bucket (advisory)

Trigger: either (a) computed crosswind exceeds the applicable limit, and/or (b) the WX suggests a likely RWYCC < 3 environment.

Why RWYCC is “likely”: METAR/TAF do not report RWYCC. The monitor infers a conservative bucket from WX tokens (e.g. freezing rain / ice pellets / heavy snow). Treat this as a heads-up; confirm with SNOWTAM/RWY reports.

Evidence: the Drawer shows the inference path: matched WX tokens → estimated RWYCC bucket → limit used.

COLD CORR OM Cold temperature corrections indicator

Trigger: temperature is low enough that cold‑temperature altitude corrections may apply (based on OM/operational practice).

Why: in very cold conditions, procedure/altimetry corrections become relevant. The monitor flags it as a reminder to check the applicable guidance.

4.5 Notifications & attention mechanisms

AUTO browser notification UX Only for new METAR + newly impacted BASE airport

When it fires (AUTO view only):

  • AUTO view is active (TV view does not use browser notifications),
  • Notifications are enabled via the UI button,
  • a new METAR arrived (new observation time),
  • and the set of impacted BASE airports (NOW/METAR triggers) contains at least one airport that was not impacted previously.

What counts as a BASE? Airports listed in base.txt (station role = BASE).

Why so strict? To avoid noise. Only “newly impacted” bases get a popup; repeated alerts are not spammed.

TV view: NEW ALERTS toast + tile flash UX Visible but not continuous

What it does: in TV view, when new alerts appear, a short “NEW ALERTS” overlay list (up to 10 lines) is shown, and relevant tiles flash for a limited time. Goal: changes are noticed without constant “siren” behaviour.

TAF panel collapsed? If a new alert comes from FORECAST and the panel is collapsed, the TAF summary tile flashes too.

5) OM quick glossary & abbreviations

Glossary

This section is a quick-reference glossary for the most common abbreviations used in the monitor. It is intentionally practical: it explains what a term usually means and how it appears in METAR/TAF and/or OM discussions.

Core meteorology / reporting terms

Term Meaning (plain language)
METAR Routine/current aerodrome weather observation. Used for NOW tiles (METAR‑primary).
TAF Aerodrome forecast covering a time window. Used for FORECAST tiles (TAF‑only, separate panel).
VIS / CAVOK Reported visibility. CAVOK implies good visibility and no significant cloud below a threshold (per METAR rules).
RVR (e.g. R04/0300) Runway Visual Range. May be reported as TDZ/MID/END. In low vis, RVR is treated as the stronger operational signal than VIS.
CMV Ceiling and Maximum Visibility. In some contexts, OM/procedures reference CMV for approach/continuation logic.
CIG / SCT/BKN/OVC Ceiling and cloud layers. BKN/OVC define ceiling; low ceiling elevates risk and score.
CB / TCU Cumulonimbus / Towering cumulus. Appears in cloud groups (e.g. BKN020CB, SCT030TCU). CB/TCU can elevate the score; TS tile requires a TS/VCTS‑type WX token.
TEMPO, BECMG, PROB30/40, NSW TAF change groups: temporary / becoming / probability / no significant weather. Always check timing and duration when interpreting FORECAST tiles.

WX codes frequently used by the monitor

Code Plain meaning
TS, VCTSThunderstorm / vicinity thunderstorm. Drives the TS/CB tile.
RA, DZRain / drizzle. Can affect inferred runway condition and score.
SN, SHSN, BLSNSnow / snow shower / blowing snow. Drives the SNOW tile; may also affect inferred RWY condition.
FZRA, FZDZFreezing rain / freezing drizzle. High impact; used in OM prohibition / RWY condition inference.
PL, GS, SGIce pellets / small hail or snow pellets / snow grains. Relevant for OM “heavy precip” and runway condition inference.
GRHail. High impact; included in OM prohibition detection.
FZFGFreezing fog. With very low VIS it triggers the ENG ICE OPS tile.
FC, SQ, WSFunnel cloud/tornado, squall, wind shear. High impact; included in OM-related hazard logic.
VAVolcanic ash. Always treat as high-impact and verify with official sources.

Operational terms

Term Meaning (plain language)
LVTO Low Visibility Take-Off. In this monitor, LVTO-related tags are shown when the reference RVR/VIS drops below defined bands (e.g. <550, <400, <150, <125).
LVP Low Visibility Procedures. Aerodrome/ATC status; the monitor uses an “LVP (<400)” advisory band as a heads-up.
CAT I/II/III, DH ILS/approach categories and decision height. Exact minima depend on approvals/equipment/lighting; the monitor uses generalised RVR bands for awareness.
RWYCC Runway Condition Code (0–6). Not available from METAR/TAF; the monitor can only infer a conservative “likely” bucket from WX tokens (advisory). Confirm with SNOWTAM/RWY reports.
SNOWTAM Runway surface condition report. Required to confirm actual RWYCC/contamination; not provided by METAR/TAF.
XWIND Crosswind component. The monitor computes it from wind and the best runway heading (if runway data exists), then compares to an OM-derived limit (advisory).
BASE airport Airports listed in base.txt. Used to reduce noise for notifications: browser popups fire only for newly impacted BASE airports in AUTO mode.

6) FAQ / troubleshooting

Help
Why is “Worst visibility” lower than the METAR VIS? Conservative spotlight by design

“Worst visibility” is intentionally conservative. It may take the minimum across reported RVR groups and/or include other conservative logic. Tiles are split: NOW is METAR‑primary; FORECAST is TAF‑only. When in doubt, validate with raw METAR/TAF and the Drawer evidence.

Why doesn’t TS/CB trigger on CB/TCU alone? TS token vs convective cloud

The TS tile is triggered by explicit TS-type WX tokens (TS, -TSRA, +TSGR, VCTS…). CB/TCU (e.g. BKN020CB) increases risk and can elevate the score, but on its own does not always mean a thunderstorm is present. This reduces false positives while still acknowledging convective potential.

Why is XWIND not calculated for an airport? Missing runway geometry data

XWIND requires runway headings (and ideally widths for “narrow runway” logic). If runway data is missing or incomplete for a station, the monitor cannot compute a reliable crosswind component. The Drawer “Why?” panel will explicitly state what data is missing.

Why don’t I get browser popups (AUTO)? Permission + strict conditions

Browser notifications require user permission (Notification permission). In addition, the monitor is intentionally strict: it fires only in AUTO view, only when notifications are enabled in the UI, only on a new METAR, and only when a newly impacted BASE airport appears compared to the previous update.

“Last update” is old. Why doesn’t it refresh? GitHub Actions / Pages pipeline

The dashboard is static HTML. Updates happen when GitHub Actions generates a new data/latest.json and deploys to Pages. If Actions is disabled, rate-limited, failing, or the schedule is off, the site will stay on the last successful run. Check: Repo Settings → Pages → Build and deployment = GitHub Actions, and the Actions tab for recent runs.
